# コンテンツスタイルガイド

このガイドは、リポジトリ内のすべてのコンテンツファイル — スキル（`skills/*/SKILL.md`）、エージェント（`agents/*.md`）、チーム（`teams/*.md`）、ガイド（`guides/*.md`） — に対する標準的な **markdown フォーマット** 規約を定義します。4 つの `_template.md` ファイルは *構造*（必須セクション、フロントマターのフィールド）を扱いますが、このガイドは *フォーマット*（それらのセクション内の markdown をどう書くか）を扱います。

以下のルールは **既存の主流スタイル** を成文化したもので、コーパスが一貫して読め、消費されるあらゆる場所 — GitHub、静的サイト、CLI ページレンダラー — でクリーンにレンダリングされるようにします。これらは理想論ではありません。ある規約がすでにほぼ普遍的である場合（先頭パイプ付きのテーブル、`-` の箇条書き）、このガイドは新しい規約を発明するのではなく、それを記録します。

## このガイドを使うとき

- 新しいスキル、エージェント、チーム、またはガイドを作成し、最初からフォーマットを正しくしたいとき
- コンテンツの PR をレビューし、客観的なフォーマットの参照が必要なとき
- 既存のファイルを編集していて、どのテーブルやフェンスのスタイルを使うべきか不明なとき
- コンテンツのフォーマットを検証する CI チェックを書く、または拡張するとき
- 「どのスタイルが正しいか」という意見の相違を解決するとき

## 前提条件

- [GitHub-Flavored Markdown](https://github.github.com/gfm/)（GFM）に精通していること
- 作成するコンテンツタイプに対応する `_template.md`（構造が先、このガイドはフォーマットを洗練させる）
- スキルの場合: [create-skill](../skills/create-skill/SKILL.md) と [creating-skills](creating-skills.md)
- エージェントとチームの場合: [agent-best-practices](agent-best-practices.md) と [creating-agents-and-teams](creating-agents-and-teams.md)

## ワークフロー概要

これらの規約は、後からのクリーンアップとしてではなく、執筆中に適用してください。CI チェック（`validate-content-style.yml`）は、コンテンツ PR に対して機械的に検出可能なルールを強制します — [適用の強制](#適用の強制)を参照。構造（フロントマター、必須セクション）はテンプレートとタイプ固有のバリデーターが所有します。このガイドは、その構造の内側にある markdown フォーマットを所有します。

## テーブル

GFM のパイプテーブルを使います。3 つのルールがあり、いずれもすでにコーパスの大多数で守られています。

1. **コンパクトな区切り行** — 列ごとにちょうど 3 個のダッシュ: `|---|---|---|`。ダッシュを列幅に合わせてパディングし **ない** こと。レンダリング出力はダッシュの数を無視するため、幅を合わせた「装飾的な」区切りはソースを肥大化させ、ノイズの多い差分を生むだけです。
2. **常に先頭と末尾のパイプを使う** — `| a | b |` であり、`a | b` ではありません。どちらも GFM 上は正当ですが、先頭パイプ形式がコーパスの標準です。
3. **意味があるときだけアラインメントマーカーを使う** — `:---`（左）、`---:`（右）、または `:---:`（中央）は、アラインメントが意味を持つとき（例えば数値列を右寄せするとき）にのみ追加します。それ以外の場合は省略します。

良い例:

```markdown
| Input | Type | Description |
|---|---|---|
| Layout spec | Configuration | Canvas dimensions and margins |
| Style params | CSS | Colors, fonts, stroke widths |
```

悪い例:

```markdown
| Input        | Type          | Description                    |
|--------------|---------------|--------------------------------|
| Layout spec  | Configuration | Canvas dimensions and margins  |
Style params | CSS | Colors, fonts, stroke widths
```

「悪い例」は両方のアンチパターンを示しています。幅を合わせた装飾的なダッシュ（2 行目）と、先頭パイプの欠落（4 行目）です。

## コードフェンス

- 開始フェンスでは **常に言語をタグ付け** します: `bash`、`r`、`yaml`、`json`、`python`、`markdown`、`text`、`console`、`diff`。どの言語も当てはまらないときは `text`（プロンプトや出力を伴うシェルセッションには `console`）を使います — タグを空のままにしないこと。
- インデント（4 スペース）のコードブロックではなく、**フェンス付きブロック**（` ``` `）を使います。フェンス付きブロックは言語タグを持ち、曖昧さがありません。
- **R コード** はパッケージ修飾付きの呼び出しを使います — `library(devtools); check()` ではなく `devtools::check()` — リポジトリ全体の R 規約に従います。このガイドは R のルールを再掲しません。[creating-skills](creating-skills.md) を参照してください。
- フェンス付きブロックを例の *内側* に示すには、内側のトリプルバッククォートがそのままレンダリングされるよう、例を 4 個のバッククォートのフェンスで囲みます。

良い例:

```bash
npm run update-readmes
```

悪い例（タグなし — シンタックスハイライトを失い、スキャンしづらくなる）:

````markdown
```
npm run update-readmes
```
````

## 見出し

- **ATX スタイル**（`#`）、ハッシュの後にスペースを 1 つ: `## Section` であり、`##Section` や Setext の下線ではありません。
- **ファイルごとに `#`（H1）は 1 つ** だけで、ドキュメントのタイトルに予約されています。それ以外の見出しはすべて H2 以下です。
- **レベルを飛ばさない** — H2 の次は H2 または H3 であり、いきなり H4 に飛ばないこと。
- **末尾のハッシュなし** — `## Section ##` ではなく `## Section`。
- 該当する `_template.md` が規定するセクション名に合わせます。このガイドはその *名前* ではなく *形式* を統括します。

## リスト、強調、インラインコード

- **順序なしリストには `-`**（ハイフン）を使い、`*` や `+` は使いません。ハイフンは圧倒的にコーパスの標準です。
- **順序付きリストにはピリオド付きの `1.`** を使います。手順のステップには、テンプレートの番号付きステップのパターンに従います。
- **ネストしたリスト項目は、親の下にスペース 2 つでインデント** します。
- **太字** は `**text**`、*斜体* は `*text*` です。強調に `__` や `_` を使わないこと。
- **インラインコード** は、ファイルパス、コマンド、識別子、フィールド名にシングルバッククォートを使います: `skills/_registry.yml`、`total_skills`、`devtools::check()`。
- スキルの手順ステップは、スキルテンプレートが定義する `**Expected:**` / `**On failure:**` ブロックの規約を保ちます — このガイドはそれを変更しません。

## リンク

- **インラインリンク** がデフォルトです: `[label](path)`。参照スタイルのリンクは、1 つのドキュメント内で同じターゲットが何度も再利用される場合にのみ許容されます。
- リポジトリ内リンクには、ファイル自身の場所を基点とした **相対パスを使います**:
  - スキル → スキル: `[name](../other-skill/SKILL.md)`
  - ガイド → エージェント: `[name](../agents/name.md)`
  - ガイド → スキル: `[name](../skills/name/SKILL.md)`
  - ガイド → ガイド: `[name](other-guide.md)`
- リンクする前に **ターゲットが存在することを確認** します。壊れた相対リンクはコンテンツのバグです。
- むき出しの URL や「ここをクリック」ではなく、説明的なリンクテキストを使います。

## 引用ブロックとコールアウト

- `>` の引用ブロックは控えめに使います — 本物の引用や短い補足のために。
- 重いコールアウト構文よりも、既存の使い方に合わせて **太字のインラインラベル**（`**Note:**`、`**Warning:**`）による強調を優先します。

## 行とファイルの規約

- **行末は LF**（`\n`）です。リポジトリの `.gitattributes` は `*.md text eol=lf` を義務付けており、git の clean フィルタは `git add` のたびに CRLF を LF に正規化します。CRLF で執筆すると、最初のコミットでほぼファイル全体の差分（「CRLF→LF 正規化」差分）が生じます — それを避けるため LF で執筆してください。
- **すべてのファイルを末尾の改行 1 つで終える** こと。
- **フロントマター** は `---` で区切られた YAML で、テンプレートが規定する順序に保ちます。必須フィールドを並べ替えたり削除したりしないこと。
- markdown 本文に **ハードタブを使わない**。スペースを使います。
- 散文は自然な文の区切りでソフトラップします。**固定のカラム制限も、1 行 1 文のルールもありません** — 行の長さだけのために既存の段落を折り返し直さないこと。

## 適用の強制

`validate-content-style.yml` ワークフローは、`skills/`、`agents/`、`teams/`、`guides/`、または `i18n/` に触れる PR で実行されます。各ルールが差分からどれだけ確実に検出できるかによって、意図的に分けられています。

| Rule | Mode | Why |
|---|---|---|
| 装飾ダッシュの区切り | **ブロック**（追加行に対して） | 区切り行は自己識別的でコンテキストに依存しない — 行単位の検出は信頼できる |
| タグなしコードフェンス | **警告** | 開始フェンスか終了フェンスかは 1 行の差分からは区別できない。信頼できる検出にはファイル全体のフェンス状態が必要 |
| 先頭パイプの欠落 | 文書化のみ、強制せず | すでにコーパス全体で 100% 守られている |

ブロックチェックは **追加行のみ** を評価するため、レガシーファイルを編集しても、その既存の違反を正規化することを強制されません。コーパス全体の正規化 — 既存のすべてのテーブルを書き直し、約 280 ファイルに言語タグを追加すること — は別途追跡されています。それが実現すると、タグなしフェンスのチェックは、ファイル全体のフェンス状態の解析を伴って警告からブロックに昇格します。

## トラブルシューティング

| Problem | Cause | Solution |
|---|---|---|
| 1 行の編集で巨大な差分 | CRLF ブロブが `git add` で LF に正規化された | 想定どおり。PR で脚注を付ける。LF で執筆する |
| 変更していないテーブルを CI が指摘する | 追加した行がたまたま装飾的な区切りだった | その区切りを `|---|---|` に変換する。チェックされるのは追加行のみ |
| フェンス付きの例が 1 つのブロックとしてレンダリングされる | 内側の ` ``` ` がエスケープされていない | 例を 4 個のバッククォートのフェンスで囲む |
| 相対リンクが壊れている | `../` セグメントの数が間違っている | ファイル自身のディレクトリからセグメントを数える |
