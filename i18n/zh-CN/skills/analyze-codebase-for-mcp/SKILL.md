# 分析代码库以适配 MCP

扫描代码库以发现函数、REST 端点、CLI 命令和数据访问模式中适合作为 MCP 工具暴露的候选项，然后生成结构化的工具规格文档。

## 适用场景

- 为现有项目规划 MCP 服务器，需要了解应暴露什么
- 在将代码库封装为 AI 可访问工具表面之前进行审计
- 比较代码库功能与已通过 MCP 暴露的功能
- 生成工具规格文档交给 `scaffold-mcp-server` 使用
- 评估第三方库是否值得封装为 MCP 工具

## 输入

- **必需**：代码库根目录路径
- **必需**：代码库的目标语言（如 TypeScript、Python、R、Go）
- **可选**：用于比较的现有 MCP 服务器代码（差距分析）
- **可选**：领域焦点（如"数据分析"、"文件操作"、"API 集成"）
- **可选**：推荐工具的最大数量（默认：20）

## 步骤

### 第 1 步：扫描代码库结构

1.1. 使用 `Glob` 映射目录树，关注源代码目录：
   - `src/**/*.{ts,js,py,R,go,rs}` 查找源文件
   - `**/routes/**`、`**/api/**`、`**/controllers/**` 查找端点定义
   - `**/cli/**`、`**/commands/**` 查找 CLI 入口点
   - `**/package.json`、`**/setup.py`、`**/DESCRIPTION` 查找依赖元数据

1.2. 按角色分类文件：
   - **入口点**：main 文件、路由处理器、CLI 命令
   - **核心逻辑**：业务逻辑函数、算法、数据转换器
   - **数据访问**：数据库查询、文件 I/O、API 客户端
   - **工具函数**：辅助函数、格式化器、验证器

1.3. 统计总文件数、代码行数和导出符号以评估项目规模。

**预期结果：** 带有角色注释的分类文件清单。

**失败处理：** 如果代码库过大（>10,000 个文件），使用领域焦点输入缩小扫描范围到特定目录或模块。如果未找到源文件，验证根路径和语言参数。

### 第 2 步：识别暴露的函数和端点

2.1. 使用 `Grep` 查找导出函数和公共 API：
   - TypeScript/JavaScript：`export (async )?function`、`export default`、`module.exports`
   - Python：不以 `_` 为前缀的函数、`@app.route`、`@router`
   - R：NAMESPACE 中列出的函数或 `#' @export` roxygen 标签
   - Go：大写函数名（按约定导出）

2.2. 对每个候选函数提取：
   - **名称**：函数或端点名称
   - **签名**：带类型和默认值的参数
   - **返回类型**：函数产出
   - **文档**：docstrings、JSDoc、roxygen、godoc
   - **位置**：文件路径和行号

2.3. 对于 REST API，额外提取：
   - HTTP 方法和路由模式
   - 请求体 schema
   - 响应形状
   - 认证要求

2.4. 构建按潜在价值排序的候选列表（公共、有文档、有类型的函数优先）。

**预期结果：** 包含提取元数据的 20-100 个候选函数/端点列表。

**失败处理：** 如果候选项较少，扩大搜索范围包含可以公开的内部函数。如果文档稀缺，在输出中标记此风险。

### 第 3 步：评估 MCP 适用性

3.1. 对每个候选项按 MCP 工具标准评估：

   - **输入契约清晰度**：参数是否有良好的类型和文档？能否用 JSON Schema 描述？
   - **输出可预测性**：函数是否返回结构化数据（JSON 可序列化）？返回形状是否一致？
   - **副作用**：函数是否修改状态（文件、数据库、外部服务）？副作用必须明确标注。
   - **幂等性**：操作是否可安全重试？非幂等工具需要显式警告。
   - **执行时间**：能否在合理超时内完成（< 30 秒）？长时间运行的操作需要异步模式。
   - **错误处理**：是否抛出结构化错误还是静默失败？

3.2. 对每个候选项按 1-5 分评分：
   - **5**：纯函数，有类型 I/O，有文档，快速，无副作用
   - **4**：有良好类型，有文档，轻微副作用（如日志记录）
   - **3**：合理的 I/O 契约但需要封装（如返回原始对象）
   - **2**：显著副作用或不清晰的契约，需要大量适配
   - **1**：不经过重大重构不适合

3.3. 筛选 3 分及以上的候选项。将 2 分项目标记为需要重构的"未来候选项"。

**预期结果：** 带有适用性评分和理由的筛选候选列表。

**失败处理：** 如果大多数候选项低于 3 分，代码库可能需要先重构再进行 MCP 暴露。记录差距并推荐具体改进（添加类型、提取纯函数、封装副作用）。

### 第 4 步：设计工具规格

4.1. 对每个选定候选项（评分 >= 3），草拟工具规格：

```yaml
- name: tool_name
  description: >
    One-line description of what the tool does.
  source_function: module.function_name
  source_file: src/path/to/file.ts:42
  parameters:
    param_name:
      type: string | number | boolean | object | array
      description: What this parameter controls
      required: true | false
      default: value_if_optional
  returns:
    type: string | object | array
    description: What the tool returns
  side_effects:
    - description of any side effect
  estimated_latency: fast | medium | slow
  suitability_score: 5
```

4.2. 将工具分组到逻辑类别（如"数据查询"、"文件操作"、"分析"、"配置"）。

4.3. 识别工具间的依赖关系（如"list_datasets"应在"query_dataset"之前调用）。

4.4. 确定是否有工具需要封装器来：
   - 将复杂参数对象简化为扁平输入
   - 将原始返回值转换为结构化文本或 JSON
   - 添加安全防护（如数据库函数的只读封装器）

**预期结果：** 包含类别、依赖关系和封装器说明的完整 YAML 工具规格。

**失败处理：** 如果工具规格含糊，重新访问第 2 步从源代码提取更多细节。如果无法推断参数类型，标记为需要人工审查。

### 第 5 步：生成工具规格文档

5.1. 编写包含以下部分的最终规格文档：
   - **摘要**：代码库概览、语言、规模和分析日期
   - **推荐工具**：第 4 步中的完整规格，按类别分组
   - **未来候选项**：2 分项目及重构建议
   - **排除项**：1 分项目及排除理由
   - **依赖关系**：工具依赖图
   - **实现说明**：封装器需求、认证需求、传输建议

5.2. 保存为 `mcp-tool-spec.yml`（机器可读）和可选的 `mcp-tool-spec.md`（人类可读摘要）。

5.3. 如果提供了现有 MCP 服务器，包含差距分析部分：
   - 规格中有但尚未实现的工具
   - 已实现但不在规格中的工具（可能过时）
   - 实现偏离规格的工具（规格漂移）

**预期结果：** 可供 `scaffold-mcp-server` 使用的完整工具规格文档。

**失败处理：** 如果文档超出合理大小（>200 个工具），按模块拆分并交叉引用。如果代码库没有合适的候选项，生成包含重构建议的"就绪度评估"文档。

## 验证清单

- [ ] 目标代码库中的所有源文件已扫描
- [ ] 候选函数已提取名称、签名和返回类型
- [ ] 每个候选项有适用性评分和书面理由
- [ ] 工具规格包含带类型的完整参数 schema
- [ ] 每个工具的副作用已明确记录
- [ ] 输出文档是有效的 YAML（可被任何 YAML 库解析）
- [ ] 工具名称遵循 MCP 约定（snake_case，描述性，唯一）
- [ ] 类别和依赖关系形成连贯的工具表面
- [ ] 提供现有 MCP 服务器时包含差距分析
- [ ] 未来候选项部分列出 2 分项目所需的重构步骤

## 常见问题

- **暴露太多工具**：AI 助手在 10-30 个聚焦工具时效果最佳。优先考虑功能广度而非深度。避免暴露每个公共函数。
- **忽视副作用**："仅读取"但同时写日志或缓存的函数仍有副作用。使用 `Grep` 仔细审计文件写入、网络调用和数据库变更。
- **假设类型安全**：动态语言（Python、R、JavaScript）的函数可能没有类型注释。从使用模式和测试推断类型，但在规格中标记不确定性。
- **缺少认证上下文**：在认证 Web 请求中工作的函数通过 MCP 调用时可能因缺少会话上下文而失败。检查隐式认证依赖，如会话 cookie、JWT 令牌或环境注入的凭证。
- **过度工程封装器**：如果函数需要 50 行封装器才能兼容 MCP，它可能不是好的候选项。优先选择自然映射到工具接口的函数。
- **忽略错误路径**：MCP 工具必须返回结构化错误。抛出未类型化异常的函数需要错误处理封装器。
- **混淆内部和外部 API**：被其他内部代码调用的内部辅助函数是糟糕的 MCP 候选项。关注为外部使用设计的函数或清晰的边界 API。
- **跳过差距分析**：如果提供了现有 MCP 服务器，始终将规格与当前实现进行比较。没有差距分析，你可能会重复工作或遗漏过时工具。

## 相关技能

- `scaffold-mcp-server` - 使用输出规格生成可工作的 MCP 服务器
- `build-custom-mcp-server` - 手动服务器实现参考
- `configure-mcp-server` - 将生成的服务器连接到 Claude Code/Desktop
- `troubleshoot-mcp-connection` - 部署服务器后调试连接
- `review-software-architecture` - 工具表面设计的架构审查
- `security-audit-codebase` - 外部暴露函数前的安全审计
