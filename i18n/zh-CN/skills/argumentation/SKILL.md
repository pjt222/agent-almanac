# 构建论点

从假设经由推理到具体证据，构建严密的论点。每一个有说服力的技术主张都遵循同一三元组：清晰的假设陈述*你相信什么*，论证解释*为什么*成立，示例证明*它确实*成立。此技能教你将该结构应用于代码审查、设计决策、研究写作，以及任何主张需要论证的场景。

## 适用场景

- 撰写或审查提出技术变更的 PR 描述
- 在 ADR（架构决策记录）中论证设计决策
- 构建超越"我不喜欢这个"的代码审查反馈
- 撰写研究论点或技术提案
- 在技术讨论中挑战或捍卫某种方案

## 输入

- **必需**：需要论证的主张或立场
- **必需**：上下文（代码审查、设计决策、研究、文档）
- **可选**：受众（同行开发者、审查者、利益相关者、研究人员）
- **可选**：需要处理的反驳论点或替代立场
- **可选**：支持主张的可用证据或数据

## 步骤

### 第 1 步：制定假设

将你的主张陈述为清晰、可证伪的假设。假设不是观点或偏好——它是一个可以用证据检验的具体断言。

1. 用一句话写出主张
2. 应用可证伪性测试：有人能用证据证明这是错的吗？
3. 缩小范围：限定在特定上下文、代码库或领域内
4. 通过检查可测试标准将其与观点区分

**可证伪 vs. 不可证伪：**

| 不可证伪（观点）                | 可证伪（假设）                                                         |
|---------------------------------|------------------------------------------------------------------------|
| "这段代码很糟糕"                | "这个函数的复杂度是 O(n^2)，而 O(n) 是可以实现的"                    |
| "我们应该用 TypeScript"         | "TypeScript 的类型系统能捕获导致我们最近 6 次生产事故中 4 次的那类空引用 bug" |
| "API 设计更简洁"                | "将 5 个端点变体替换为单一参数化端点将公共 API 表面积减少 60%"        |
| "这种研究方法更好"              | "方法 A 在数据集 X 上以 95% 置信水平比方法 B 达到更高精度"           |

**预期结果：** 一句具体、有范围、可证伪的假设。读者能立即设想什么证据会确认或反驳它。

**失败处理：** 若假设感觉模糊，应用"我如何反驳这个？"测试。若你无法设想反证，该主张是观点而非假设。缩小范围或添加可衡量标准直到它变得可检验。

### 第 2 步：识别论证类型

选择最能支持假设的逻辑结构。不同的主张需要不同的推理策略。

1. 回顾四种论证类型：

| 类型 | 结构 | 最适合 |
|------|------|--------|
| 演绎 | 若 A 则 B；A 成立；因此 B | 形式证明、类型安全主张 |
| 归纳 | 在 N 个案例中观察到的模式；因此可能普遍成立 | 性能数据、测试结果 |
| 类比 | X 在相关方面与 Y 相似；Y 有属性 P；因此 X 可能有 P | 设计决策、技术选择 |
| 证据性 | 证据 E 在假设 H1 下比 H2 更可能出现；因此 H1 得到支持 | 研究发现、A/B 测试结果 |

2. 将假设与最强论证类型匹配：
   - 主张某事*必须*为真？用**演绎**
   - 主张某事*基于观察*倾向为真？用**归纳**
   - 主张某事*可能有效*基于类似先例？用**类比**
   - 主张某种解释*比替代方案更符合数据*？用**证据性**

3. 考虑组合类型以获得更强的论证（例如归纳证据支持的类比推理）

**预期结果：** 选定论证类型（或组合），并有清晰的理由说明为什么适合该假设。

**失败处理：** 若没有单一类型完全适用，假设可能需要拆分为子主张。将其分解为每个都有自然论证结构的部分。

### 第 3 步：构建论证

建立连接假设与其论证的逻辑链。

1. 陈述前提（你从中出发的事实或假设）
2. 展示逻辑联系（前提如何导向结论）
3. 加固最强的反驳论点：在反驳之前以最佳形式陈述最强的反对意见
4. 用证据或推理直接回应反驳

**实例——代码审查（演绎 + 归纳）：**

> **假设**：「将验证逻辑提取到共享模块将减少三个 API 处理器中的 bug 重复。」
>
> **前提**：
> - 三个处理器（`createUser`、`updateUser`、`deleteUser`）各自以细微差异实现相同的输入验证（在 `src/handlers/` 中观察到）
> - 在过去 6 个月中，5 个验证 bug 中有 3 个在一个处理器中修复但未传播到其他处理器（见问题 #42、#57、#61）
> - 共享模块为逻辑强制实施单一事实来源（演绎：若只有一个实现，则只有一处需要修复）
>
> **逻辑链**：因为三个处理器复制了相同的验证（前提 1），在一个中修复的 bug 在其他地方被遗漏（前提 2，从 3/5 案例归纳）。共享模块意味着修复一次适用于所有调用者（从共享模块语义演绎）。因此，提取将减少 bug 重复。
>
> **反驳（加固）**：「共享模块引入耦合——对一个处理器验证的更改可能破坏其他处理器。」
>
> **反驳回应**：处理器已经共享相同的验证*意图*；耦合是隐式的，更难维护。通过带参数化选项的共享模块（例如 `validate(input, { requireEmail: true })`）使其显式，使耦合可见且可测试。当前的隐式重复更危险，因为它隐藏了依赖关系。

**预期结果：** 完整的论证链，包含前提、逻辑联系、加固的反驳论点和回应。读者能逐步跟随推理。

**失败处理：** 若论证感觉薄弱，检查前提。薄弱论证通常源于无支持的前提，而非错误的逻辑。为每个前提找到证据或承认它是假设。若反驳比回应更强，假设可能需要修订。

### 第 4 步：提供具体示例

用独立可验证的证据支持论证。示例不是插图——它们是使论证可检验的实证基础。

1. 提供至少一个**肯定示例**来确认假设
2. 提供至少一个**边界情况或极端示例**来测试限制
3. 确保每个示例**独立可验证**：另一个人可以在不依赖你的解释的情况下重现或检查
4. 对于代码主张，引用具体文件、行号或提交
5. 对于研究主张，引用具体论文、数据集或实验结果

**示例选择标准：**

| 标准 | 好的示例 | 差的示例 |
|------|---------|---------|
| 独立可验证 | "问题 #42 显示 bug 在处理器 A 中修复但不在 B 中" | "我们以前见过这类 bug" |
| 具体 | "`createUser` 第 47 行重新实现了与 `updateUser` 第 23 行相同的正则表达式" | "代码库中有重复" |
| 代表性 | "过去 6 个月中 5 个验证 bug 中有 3 个遵循此模式" | "我曾见过一个类似的 bug" |
| 包含边界情况 | "此模式适用于字符串输入，但不适用于文件上传验证，后者有处理器特定的约束" | （未提及任何限制） |

**预期结果：** 读者可以独立验证的具体示例。至少一个肯定示例和一个边界情况。每个引用具体工件（文件、行、问题、论文、数据集）。

**失败处理：** 若示例难以找到，假设可能太宽泛或没有扎根于可观察现实。缩小范围到你实际能指向的内容。缺少示例是信号，不是要用模糊引用来掩盖的缺口。

### 第 5 步：组合完整论点

将假设、论证和示例组合成适合上下文的格式。

1. **对于代码审查**——将评论结构为：
   ```
   [S] <建议的一行摘要>

   **Hypothesis**: <你相信应该改变什么及为什么>

   **Argument**: <逻辑论证，包含前提>

   **Evidence**: <具体文件、行、问题或指标>

   **Suggestion**: <具体代码变更或方法>
   ```

2. **对于 PR 描述**——将主体结构为：
   ```markdown
   ## Why

   <假设：此变更解决的问题和具体的改进主张>

   ## Approach

   <论证：为什么选择此方法而非替代方案>

   ## Evidence

   <示例：基准测试、bug 引用、前后对比>
   ```

3. **对于 ADR（架构决策记录）**——使用标准 ADR 格式，三元组映射到 Context（假设）、Decision（论证）和 Consequences（示例/预期结果证据）

4. **对于研究写作**——映射到标准结构：Introduction 陈述假设，Methods/Results 提供论证和示例，Discussion 处理反驳

5. 审查组合的论点：
   - 逻辑缺口（结论是否真的从前提得出？）
   - 缺失证据（是否有无支持的前提？）
   - 未处理的反驳（是否回答了最强的反对意见？）
   - 范围蔓延（论证是否保持在假设范围内？）

**预期结果：** 一个完整的、适合其上下文的格式化论点。读者可以评估假设、跟随推理、检查证据并考虑反驳——所有这些都在一个连贯的结构中。

**失败处理：** 若组合的论点感觉脱节，假设可能太宽泛。将其拆分为聚焦的子论点，每个都有自己的假设-论证-示例三元组。两个紧凑的论点比一个冗长的更强。

## 验证清单

- [ ] 假设是可证伪的（有人可以用证据反驳）
- [ ] 假设有范围限定，而非普遍主张
- [ ] 论证类型已识别且适合该主张
- [ ] 前提明确陈述，而非假定为共同知识
- [ ] 逻辑链连接前提到结论，无缺口
- [ ] 最强的反驳被加固并得到回应
- [ ] 至少一个肯定示例支持假设
- [ ] 至少一个边界情况或限制被承认
- [ ] 所有示例独立可验证（已提供引用）
- [ ] 输出格式与上下文匹配（代码审查、PR、ADR、研究）
- [ ] 无逻辑谬误（诉诸权威、虚假二分法、稻草人）

## 常见问题

- **将观点陈述为假设**：「这段代码很乱」是偏好，不是假设。改写为可检验的主张：「此模块有 4 个职责应按单一职责原则分离，其 6 个跨越 3 个不相关领域的公共方法为证。」
- **跳过反驳**：未处理的反对意见会削弱论证，即使读者从未说出来。始终加固——在反驳之前以最佳形式陈述最强的反对意见。
- **示例模糊**：「我们以前见过这种模式」不是证据。指向具体问题、提交、行、论文或数据集。若找不到具体示例，假设可能没有很好地扎根。
- **诉诸权威**：「高级工程师这么说」或「Google 这样做」不是逻辑论证。权威可以*激励*调查，但论证必须独立站得住脚。
- **结论中的范围蔓延**：得出比证据支持的更宽泛的结论。若示例涵盖 3 个 API 处理器，不要得出关于整个代码库的结论。使结论范围与证据范围匹配。
- **混淆论证类型**：对演绎主张（「必须是」）使用归纳语言（「倾向于是」），或反之。对结论的强度要精确——演绎论证给出确定性，归纳论证给出概率。
